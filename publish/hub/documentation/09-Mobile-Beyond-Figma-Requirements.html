NutriSync Mobile — Requirements beyond the Figma

Scope the Figma screens do not cover but the briefing commits to. Two families: (A) health & sport connectors, and (B) data rights + account lifecycle — plus (C) the cross-cutting mobile capabilities those imply. Grounded in the Developer Specification (§6 CAS, §13 Settings) and the business-case / deck roadmap (wearables strategy, M4→Y3).

A. Health, wearable & sport connectors

Why it matters to the product, not just "integrations": the CAS algorithm today is fed by manual logs — the morning mood/energy gate, Edit Period, and MovementLog (Dev Spec §6.1). Connectors let those same signals arrive automatically and objectively, which is exactly what the deck's "data moat" and "50% churn reduction for connected users" depend on. A1. OS-native health MVP+ · M4–6

  • iOS: Apple Health (HealthKit) — the primary iOS integration. Reads data the Apple Watch writes into Apple Health (HRV, sleep, skin temperature, workouts, blood oxygen) plus iPhone data. Request read scopes per category with granular consent; use observer queries + background delivery so new samples sync without opening the app. Optionally write NutriSync's predicted period/phase back to Apple Health.
  • Android: Health Connect (Google Fit is deprecating) — equivalent read permissions and background sync.
  • Auto-map: energy proxy ← HRV + sleep; MovementLog activities ← workouts; Phase Detection confidence ← skin/wrist temperature trend.

A2. Dedicated wearables Y1–Y3

  • Oura and Garmin via their partner/health APIs (OAuth2); Fitbit/Whoop optional.
  • Cloud-to-cloud pull on a schedule + webhook where offered; store only derived metrics, not the raw firehose.

A3. Sport / activity apps

  • Strava (OAuth2 + webhook) and native workouts → auto-populate the MovementLog checklist (intensity tags already defined in movement_activity). User confirms/edits.

A4. CGM Series A · Y3

  • Dexcom and Abbott Libre APIs — glucose curve as an advanced biomarker layer.

A5. Connector UX & technical requirements (all new screens)

  • "Connections / Integrations" settings screen (not in Figma): providers list, connect/disconnect, last-sync time, per-metric toggles, connection health/errors.
  • Optional onboarding step: "Connect a wearable" (skippable) after Consent.
  • Permission priming screens before the OS health prompt (raise grant rates).
  • Secure token storage (Keychain / Keystore), token refresh, revoke on disconnect.
  • Background sync scheduling; offline queue; de-dup and conflict rules (manual log vs device).
  • Attribution/trust rule: device data augments but never silently overrides a user's manual log — show source badges.

B. Data rights & account lifecycle GDPR · health = special category

The Dev Spec names these in §13.5 but they need full flows, backend, and legal handling. All are MVP / launch-blocking for an EU health app.

B1. Export / "Download my data" (Dev Spec §13.5)

  • Full machine-readable export (JSON + CSV) of everything: profile, cycles, all four daily logs, biomarkers, CAS history, preferences, and connector-derived data.
  • Async job → secure, expiring download link delivered in-app + by email.
  • Covers GDPR right to access and data portability.

B2. Delete data / Delete account (Dev Spec §13.5 — "requires confirmation")

  • Hard delete with explicit confirmation (type-to-confirm), then a short grace/undo window.
  • Cascade: erase all Supabase rows (RLS-scoped), revoke all third-party grants (HealthKit authorization, Strava/Oura/Garmin tokens), purge exports and backups per retention policy.
  • Apple App Store requires in-app account deletion — this is a submission gate, not optional.

B3. Close / deactivate account (soft)

  • Offer deactivate (reversible, pauses data processing) as a lighter step distinct from delete — reduces accidental hard-deletes and churn-by-mistake.

B4. Consent & processing controls (Dev Spec §13.5, §3.8)

  • Granular, withdrawable toggles: Analytics & crash reports / Personalised insights / Research participation. Withdrawal takes effect immediately and propagates to processors.
  • Activity log ("Manage activity log"), cookie/tracking preferences, view Privacy Policy & Terms.
  • iOS App Tracking Transparency prompt; keep HealthKit data out of any ads/analytics per Apple's HealthKit policy (no selling/advertising use of health data).

B5. Rectification & transparency

  • Edit personal info (Dev Spec §13.2) satisfies right to rectification.
  • Clear, per-connector disclosure of what's collected and why (layered privacy notice).

C. Cross-cutting mobile capabilities implied (also absent from Figma)

  • Auth hardening (Dev Spec §13.4): TOTP 2FA, passkeys / biometric unlock (Face ID / Touch ID), active-sessions list with remote log-out per device.
  • Push notifications & reminders (Dev Spec §13.1): permission priming, daily log reminders, phase-change nudges, quiet hours.
  • Permissions matrix: Health, Notifications, Location (city), Motion, ATT — each with a rationale screen and graceful denial handling.
  • Offline-first sync: local cache + write queue so the real-time CAS keeps working without signal; reconcile on reconnect.
  • Health-data compliance & security: EU data residency, encryption in transit + at rest, HealthKit/Health Connect entitlements, GDPR special-category safeguards, DPA with processors.
  • Localisation: EN + ES at minimum (beta testers in Greece & Spain), metric/imperial units.
  • Accessibility: dynamic type, VoiceOver/TalkBack labels, contrast.

Suggested priority for the build

MVP / launch-blocking: HealthKit + Health Connect (A1), Data export (B1), Delete + in-app account deletion (B2), Consent controls (B4), 2FA + biometric unlock (C), notifications (C).

Fast-follow (Y1): Strava + Garmin (A3/A2), Connections screen polish (A5), deactivate (B3), passkeys, offline sync hardening.

Later (Y2–Y3, per deck): Oura (A2), CGM (A4), write-back to HealthKit, research-data pipeline.

What this adds to the app we're building

New screens/modules not in the current Figma set, to add to the Expo app: Connections (integrations list + per-provider detail), permission-priming screens, an onboarding "Connect a wearable" step, and the Settings Data Privacy flow (export, delete, deactivate, consent toggles, activity log) + Sign-in & Security (2FA, passkeys, sessions). These can be wired alongside the Figma screens as the build progresses.
