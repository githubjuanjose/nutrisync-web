NutriSync — Developer Spec → Implementation Plan & Coverage

Source: NutriSync Collective — Complete Product & Developer Specification, v1.0 (June 2026), 29 pages. This plan extracts the user stories/functionality, records what is already built in this repo, and lays out the build waves to reach a fully operational app. The spec is the single source of truth; where it conflicts with earlier inferences (e.g. luteal colour), the spec wins.

Status legend: ✅ done · 🟡 partial/foundation · ⬜ to build.

0. Corrections the spec forced (apply everywhere)

  • Brand primary is #E8472A (not #EA5740); background cream #F5F2EB. → updated in apps/mobile/src/theme.ts and packages/core/tokens.mjs.
  • Phase colours (authoritative): Menstrual #E8472A · Follicular #6B9E6B (sage) · Ovulatory #D4A017 (gold) · Luteal #7B5EA7 (purple/lavender). My earlier prototypes showed luteal as pink — now corrected in tokens.
  • Five scoring phases, not four: menstrual, follicular, ovulatory, early_luteal, late_luteal. → packages/core/cas.mjs.
  • Bottom nav: Calendar · Meals (NutriLog) · Movement (MovementLog) · Progress (+ Home). Earlier app used Home/Meals/Progress/Community/Profile — realign.
  • The Nutri mascot (round orange character, 4 avatar variants) is a first-class brand element — must be added.

2. User stories by module → status

Onboarding (§3) ⬜

  • As a new user I create an account and name my Nutri, so the app feels personal. ⬜
  • As a user I answer Cycle Info (8 steps) incl. last period start date (the critical field), so phases can be calculated. ⬜ *(spec flags the date as missing from wireframe but required)*
  • As a user I provide Health (4), Nutrition prefs (4), NutriGoal, Nutri avatar, city, and consent, so every recommendation is filtered to me. ⬜
  • As a user my answers cascade (vegan removes salmon, nut allergy removes almonds, PCOS prioritises low-GI, etc., §3.9) at the data layer. ⬜
  • Acceptance: no health row persists without consent; BMR computed via Mifflin-St Jeor but never shown as calories (§3.4). ⬜

Daily Mood & Energy gate (§4) ⬜

  • As a user, each calendar day on first open I log mood then energy on vertical pill sliders, feeding CAS C2 & C5. ⬜ (show once/day; check if already logged.)

Home (§5) 🟡→⬜

  • As a user I see today's phase + cycle day ring (arc colour = phase colour), greeting, week strip, edit period + edit health, bottom nav. 🟡 (ring exists; needs phase colours, cycle-synced badge, IA, edit-health.)
  • cycle_day = today − last_period_start + 1, auto, never manual (§5.2). ✅ in cas.mjs.

CAS (§6) ✅ engine / ⬜ UI wiring

  • As a user my CAS (0–100) updates in real time as I log (§15). ✅ engine; ⬜ wire writes → recompute → daily_scores upsert.
  • CSS (Cycle Stability Score, §6.8) — donut on Progress; MVP = rolling avg of last 3 cycle CAS averages; full logic post-MVP. ⬜ (mark future in code.)

NutriLog / Meals (§7) 🟡→⬜

  • As a user I see Daily Tip / Body Insight that change every day (indexed by cycle_day, modulated by NutriGoal), a Nutri Basics checklist pre-checked, and a TODAY'S MEAL free-text log. ⬜
  • Food lists filtered by diet/allergies/conditions before display (§7.3). ⬜ Checking recalculates C3 in real time. ⬜
  • Full phase food content (§7.4) → seed nutrition_checklist content library. ⬜

MovementLog (§8) ⬜

  • As a user I see phase movement tips + a 9-item checklist (pre-checked) + free-text log; highest-intensity checked drives C4 + C2 sub-5. ⬜

Edit Period (§9) ⬜

  • As a user I log flow, mood-state, pain, digestion, breasts, cravings, skin, sleep, energy, body changes, repro, libido, birth control, sex log, notes — feeding biomarkers + pattern recognition. ⬜
  • Sleep→C2-sub3 mapping (§9.3), cravings→appetite (§9.4), sex log → calendar star (■ protected / ★ unprotected, private). ⬜
  • Pattern recognition after 2 cycles → Progress insights (§9.5). ⬜

Edit Health / NutriGoals (§10) ⬜

  • As a user I view/edit allergies & conditions; my NutriGoal is locked until completed_cycles ≥ 2 (§10.2). ⬜

Calendar (§11) 🟡→⬜

  • Today banner + phase pill + month grid with phase-colour dots, dashed predicted period, legend, selected-day insight, sex-log stars, Cycle History year view. ⬜
  • Tap past/today → Edit Period prefilled; future → predictions, no logging (§11.4). ⬜

Progress (§12) 🟡→⬜

  • CAS summary card + 5 mini component rings; CSS donut; CAS History (weekly/monthly/yearly, line chart, score-by-phase bars, contributors); biomarker snapshot; cycle insights. ⬜ (current Progress screen is a simpler placeholder.)

Settings (§13) 🟡→⬜

  • Personal info, Nutri avatar picker, sign-in & security (2FA, sessions), data privacy (download/delete, toggles), community privacy, send feedback. 🟡 (profile/privacy stub exists.)

Personalisation engine (§3.9, §7.3, §16) ⬜

  • As any user, dietary restrictions/allergies/conditions/NutriGoal filter and modulate all food, movement, tip, and insight content at the data layer. ⬜ (rules documented; implement as a content-filter service.)

Longitudinal intelligence (§6.3, §6.8, §9.5) ⬜

  • After 3 full cycles, biomarker scoring switches from population norms to the user's own per-phase averages (user_phase_averages). ⬜ (schema ready; logic to build.)

3. Non-negotiable rules (§16) — implement as guardrails/tests

  1. Phase determined before score; no last_period_start_date → C1 = 0 and CAS blocked. ✅ (engine + test)
  2. Never penalise aligned symptoms (low energy menstrual, high appetite luteal, fragmented sleep/variable mood late-luteal = full marks). ✅ (engine + test)
  3. Checklists default checked; user unchecks what they didn't do. ⬜ (UI)
  4. Fitness opt-out → 10/20 passive. ✅ (engine)
  5. NutriGoal locked 2 cycles. ⬜ (UI + completed_cycles)
  6. All data user-scoped, RLS on every table. ✅ (schema)
  7. Sex-log stars (■/★), calendar-only, private. ⬜
  8. CSS ≠ CAS — daily alignment vs cross-cycle stability. (documented; CSS post-MVP)
  9. Daily content changes every day within a phase (indexed by cycle_day). ⬜ (content model)
  10. Personalisation overrides phase defaults at the data layer. ⬜
  11. Cycle-length personalisation (not fixed 28). ✅ (engine)
  12. After 3 cycles → personal averages. ⬜
  13. BMR internal only — never show calories. ⬜ (enforce in UI)
  14. NutriGoal modulates every tip/insight/food. ⬜

4. Build waves (recommended)

Wave A — Foundations (done/this pass): ✅ CAS engine + tests · ✅ tokens · ✅ schema. Next: wire CAS recompute Edge Function (§15) and the content-filter service (§3.9).

Wave B — Core daily loop (P0): Onboarding (account → Nutri → cycle date → prefs → consent) · Mood/Energy gate · Home (phase ring, correct colours, edit period/health) · NutriLog (tips + pre-checked checklist + meal log + C3 wiring) · MovementLog (checklist + C4 wiring) · live CAS on Home/Progress.

Wave C — Depth (P1): Edit Period full logging + sex-log stars · Calendar (month grid, predictions, history) · Progress (CAS history, CSS donut, contributors, biomarker snapshot) · Settings (avatar, security, privacy, feedback).

Wave D — Intelligence (P1→P2): Pattern recognition (2 cycles) · personal per-phase averages (3 cycles) · full CSS computation · NutriGoal modulation depth.

Cross-cutting: every screen consumes packages/core tokens + cas.mjs; food/movement content seeded from §7.4 / §8.3; all writes go through RLS; analytics events feed the admin MIS.
