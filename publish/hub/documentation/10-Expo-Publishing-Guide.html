NutriSync — Expo build & publishing guide

Everything needed to test the app on real phones and publish it to the App Store and Google Play, using Expo Application Services (EAS). Written for the current Expo SDK 51 project in /mobile.

0. Accounts you need (once)

Account Cost Why
Expo account Free Runs EAS Build/Submit/Update. Sign up at expo.dev.
Apple Developer Program $99 / year Required to build a real iOS app, use TestFlight, and ship to the App Store.
Google Play Console $25 one-time Required to publish on Android / Play internal testing.

You can do all testing (Expo Go + internal builds) before paying for the stores; the fees are only needed for TestFlight / store distribution.

1. Tools

npm install -g eas-cli
eas login            # sign in to your Expo account
cd mobile
npm install

Link the project to your Expo account (creates the EAS project + writes the id into app config):

eas init

2. Fastest path — test today with Expo Go (no build, no fees)

cd mobile
npx expo start

Scan the QR with the Expo Go app (iOS App Store / Google Play). Live reload on save. This is how the team in Greece & Spain can try it immediately — send them the QR or the exp:// link. Limitation: Expo Go can't run custom native modules (e.g. HealthKit), so once we add wearable connectors you'll move to a development build (next section).

3. Development build (internal, for native features + device testing)

A development build is your own app binary with the dev tools baked in — needed once we add native modules (HealthKit, notifications, biometrics).

# iOS (installs to registered devices or a simulator)
eas build --profile development --platform ios

# Android (installs an .apk on any device)
eas build --profile development --platform android

EAS returns a URL/QR; open it on the phone to install. Then run npx expo start --dev-client and the build connects to your Metro bundler.

For iOS physical devices you must register them first:

eas device:create        # sends a link the tester opens on their iPhone

4. Preview build (shareable, no store) — best for beta testers

Gives a standalone install your co-founders can run without Expo Go or Metro.

eas build --profile preview --platform all
  • Android: produces an .apk — share the link, tap to install.
  • iOS: internal distribution requires the testers' devices registered on your Apple account (eas device:create), or use TestFlight (section 6) which is smoother for iOS.

5. Production build (store-ready binaries)

eas build --profile production --platform all

Produces an .aab (Android App Bundle) and a .ipa (iOS). autoIncrement bumps the build number each time. Credentials (signing keys, provisioning) are generated and stored by EAS — say yes when it offers to manage them.

6. Submit to the stores — eas submit

Fill the placeholders in mobile/eas.json → submit.production first (Apple ID, App Store Connect app id, Apple Team id; Android service-account JSON).

# iOS → App Store Connect / TestFlight
eas submit --profile production --platform ios --latest

# Android → Play Console (internal track by default)
eas submit --profile production --platform android --latest

iOS first-time setup

  1. In App Store Connect create the app (bundle id com.nutrisync.app).
  2. eas submit uploads the build; it appears in TestFlight after Apple processing (~15 min).
  3. Add your team as internal testers in TestFlight → they get the app on their iPhones. This is the recommended way to get the Greece/Spain team on iOS.

Android first-time setup

  1. In Play Console create the app.
  2. Create a service account (Google Cloud → grant it Play Console access), download the JSON, save as mobile/google-service-account.json (git-ignored).
  3. eas submit pushes to the internal testing track → add testers by email; they install from a Play link.

7. Over-the-air (OTA) updates — ship JS fixes without a new store build

Because this is a JS/Expo app, most changes (screens, logic, copy) can go out instantly:

eas update --branch production --message "onboarding copy fix"

Installed apps pick it up on next launch — no store review. (Native changes still need a new build.) The channel fields in eas.json map builds to update branches.

8. Store listing assets & metadata (prepare in parallel)

  • App icon 1024×1024 (no alpha), splash screen — add under app.json → icon/splash.
  • Screenshots per device size (6.7" iPhone, 5.5", iPad; Android phone/tablet).
  • Name, subtitle, description, keywords, support URL, marketing URL (nutrisynccollective.com).
  • Privacy Policy URL (required — health app), age rating, data-safety form.

Health-data specifics (don't skip — these are review gates)

  • Apple: a Privacy Policy is mandatory; complete App Privacy "Nutrition/Health & Fitness" data types; provide HealthKit usage strings (NSHealthShareUsageDescription) once HealthKit is added; HealthKit data must not be used for advertising.
  • Apple requires in-app account deletion — the Settings "Delete account" flow must ship before approval (see doc 09, §B2).
  • Google Play Data safety form must declare health data collection, encryption in transit, and the deletion path.

9. Versioning

  • app.json → version is the user-facing version (e.g. 1.0.0).
  • Build numbers auto-increment via autoIncrement in the production profile.
  • Bump version for each store release; use eas update for in-between JS patches.

10. Typical release checklist

  1. npm run typecheck (or CI) is green.
  2. eas update to staging channel → smoke test on a preview build.
  3. eas build --profile production --platform all.
  4. eas submit ... --latest for both platforms.
  5. iOS: add build to a TestFlight group; Android: promote internal → closed/production.
  6. Fill/refresh store metadata, screenshots, privacy answers.
  7. Submit for review; use eas update for hotfixes after launch.
