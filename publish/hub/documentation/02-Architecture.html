NutriSync — Technical Architecture Plan

Targets: iOS, Android, and Web from one codebase, EU/CH data residency, AI personalization, pilot-ready in 2026 Q1–Q2. Principle: ship one product across mobile + web without maintaining three teams. Optimize for a 1-developer start (per GTM roadmap) that scales to a real team.

1. Recommended stack (and why)

Client — one codebase, three surfaces

Expo + React Native + react-native-web, organized as a monorepo with Expo Router.

  • Mobile (iOS/Android) is the primary surface — the deck is phone-first. React Native gives native apps.
  • react-native-web renders the same components as a responsive web app, so the marketing-to-app web experience and the mobile app share UI. The current site is a Lovable landing/waitlist page; the *product* web app is this.
  • Expo removes most native build pain (EAS Build/Submit/Update) — ideal for a lean team.

Analogy: think of the shared UI layer like a universal power adapter — one set of components, and the platform "prongs" (iOS, Android, web) just slot in.

Alternative considered: Flutter (great UI, but Dart and a smaller hiring pool); or native iOS + native Android + Next.js web (best fidelity, 3× the work — wrong for a solo start). Recommendation stands: Expo RN universal.

Backend — managed first, portable later

Phase 1 (MVP): Supabase (managed Postgres + Auth + Row-Level Security + Storage + Edge Functions), hosted in the EU region.

  • Postgres is the right home for relational cycle/log/recipe data.
  • Auth + RLS gives per-user data isolation out of the box — important for health data.
  • Edge Functions host the recommendation endpoint and any server-side AI calls (keys never touch the client).

Phase 2 (scale / B2B): introduce a dedicated API service (NestJS or FastAPI) in front of Postgres when you need complex B2B seat logic, audit logging, and the data "flywheel." Supabase stays as the DB; you graduate the business logic into your own service. This keeps you from being locked in.

AI personalization layer — a service boundary, not a hardwire

Expose recommendations behind a single internal API: POST /recommendations { userId, date } → ranked actions/meals.

  • MVP: rules-based engine over the cycle-content model (phase × profile × preferences) + lightweight ranking. Deterministic, explainable, clinician-reviewable.
  • Upgrade path: swap the engine for an LLM/ML service (the deck references Apertus AI) behind the same contract — clients don't change. Keep prompts/models server-side; log inputs/outputs for the insight loop (consented, de-identified).

Analogy: the recommender is a swappable engine in the same chassis — rules today, ML tomorrow, the dashboard (client) never knows.

Cross-cutting

  • Payments: RevenueCat over App Store / Play / Stripe — one subscription state across platforms for the €5.99 premium tier and free/premium gating.
  • Analytics/product: Amplitude or PostHog (EU) for the retention/engagement metrics in the PRD.
  • Push: Expo Notifications.
  • Errors/observability: Sentry.

3. Data model (core entities)

User(id, email, locale, created_at, subscription_tier, org_id?)
Profile(user_id, dob, goals[], dietary_prefs[], allergies[], activity_level)
CycleProfile(user_id, last_period_start, avg_cycle_len=28, avg_period_len=5)
CycleLog(id, user_id, date, type[period|symptom|mood|energy], value, notes)
  → current phase + cycle day are DERIVED from CycleProfile + latest CycleLog
Phase(key[menstrual|follicular|ovulatory|luteal], guidance, nutrition_focus, training_focus, sources[])
Recipe(id, title, kcal, prep_min, category, macros, ingredients[], steps[], phase_affinity[], evidence[])
MealLog(id, user_id, date, meal_slot, recipe_id?, kcal)
ActivityLog(id, user_id, date, steps, water_glasses, workouts[])
Recommendation(id, user_id, date, kind[meal|action], ref_id, score, rationale)
Post(id, author_id, body, topic, created_at)   Article(id, title, body, tags[], author)
Reaction(id, user_id, target_type, target_id)   Comment(...)
Org(id, name, seats, plan)  Membership(org_id, user_id, role)   // B2B
Consent(user_id, scope, granted_at)  // GDPR/FADP audit trail

Derivation rule (the heart of the app): cycle_day = (today − last_period_start) mod avg_cycle_len + 1, mapped to a Phase. All Home, Meal, and Recommendation content keys off this. Phase boundaries personalize as the user logs real data.

4. Privacy & compliance (build-time requirements)

  • Special-category data (menstrual/health) under GDPR & Swiss FADP. Required from day one: explicit, granular consent records (Consent table), data export and erasure endpoints, data minimization, EU/CH residency (Supabase EU project).
  • RLS everywhere: a user can only read/write their own rows; org admins see aggregate, never individual cycle data.
  • No selling cycle data; no third-party ad SDKs on health screens. Make "your data is yours" a visible product feature (a PRD differentiator).
  • Server-side secrets only; AI calls proxied through Edge Functions.
  • Clinical content review workflow + evidence tagging before publish.

6. Build sequence for the MVP

  1. Foundation: monorepo + Expo app shell + Supabase EU project + Auth + design tokens from Figma.
  2. Cycle engine: CycleProfile, phase derivation, Home screen with phase ring.
  3. Content: Phase + Recipe schema, seed clinician-reviewed phase content + recipe set; Meal Plans screen.
  4. Logging: meal/activity/cycle logging + Progress screen.
  5. Recommendations: rules-based service behind the /recommendations contract.
  6. Community (read + react), Profile, consent/export/erase, paywall (RevenueCat), analytics.
  7. Pilot hardening: instrumentation for the PRD metrics, employer seat stub for B2B pilots.

7. Key risks / decisions to lock

  • Apertus AI integration model (hosted vs. self-run; data residency, latency, cost).
  • Recipe content sourcing (license vs. original vs. partner) — legal + cost.
  • Tamagui vs. plain RN-web styling for the shared ui package (perf vs. simplicity).
  • When to graduate from Supabase logic into a dedicated API service (likely at B2B scale, 2027).
