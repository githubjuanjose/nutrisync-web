Design Brief — Open Requests

Fixes that live in Design's compiled web bundle (marketing index.html + web app app.html). We can only patch around these from the integration layer; Design should fix them at the source so they survive each re-export. This is the shareable version of the Open Design requests tracked in the Change Log.

Hand this straight to Design.

Status: items 1–5 closed ✅ (1–4 in the 0607v02.53 pack; item 5 error overlay root-caused in v11.1). Item 6 (Multilingual — 7 languages) is NEW and open — see below.

1 · Web app — "Back to site" exit ✅ delivered (0607v02.53)

From inside the web app journeys there's no way back to the marketing site. Add a clear, persistent control in the app header/sidebar (e.g. on the NutriSync logo, or a menu item) that returns the user to index.html. We've added a temporary floating button as a stopgap — please replace it with a proper in-header one.

2 · Web app — back buttons are too small ✅ delivered (0607v02.53)

The in-app back arrows (Edit Period, Edit Health, sub-screens) are tiny and hard to see/tap. Increase the icon size and tap target to a comfortable minimum (~24px icon, ~44px touch target).

3 · Web onboarding — Back on every step + Start over ✅ delivered (0607v02.53)

The onboarding wizard has step-back logic internally, but the Back control is missing/inconsistent on some steps. Please:

  • show Back on every step after the first;
  • add a "Start over" that resets to step 1 and clears entered answers;
  • map the browser Back button to the previous step instead of leaving the flow.

(Reference: the mobile onboarding already does exactly this — Back on all steps, hardware-Back handled, and a "Start over" with a confirm.)

4 · Hub navigation — deeper back paths ✅ delivered (0607v02.53)

The sticky room tabs + Investor↔Builder switcher (added in the 05/07 pack) work well. Remaining gap: opening a card/section and using in-page back — or the browser Back button — can still drop the user to the site landing. Please:

  • make in-page back return to the room's previous view (Overview or the card grid), never the landing;
  • wire the browser Back button to step within the hub rather than exit;
  • moving card → card → section should never force a return to Start.

5 · Remove the dev error overlay ✅ resolved (v11.1)

Root-caused rather than masked: the overlay only fires on a real asset miss, and the one genuine miss (a typo'd mood-icon, mischevious→mischievous) was fixed at source. A full asset audit confirms zero console errors on both pages. The #__bundler_err{display:none} rule is retained deliberately as a production safety net — a debug overlay should never reach users even if an asset regresses.

6 · Multilingual — 7 languages (NEW · open)

Expand from EN/ES to 7 languages: English, Spanish, French, Italian, German, Dutch, Greek.

Language selector — confirmed UX (marketing nav, web-app login, Settings card): - Two pills, always visible: [English] + [OS language]. - The second pill is the device/OS language when it's in the supported set, and it is the default selection on first load. - Edge case: if the OS language is English or unsupported → fall back to the pair [English] [Español] with English selected. - A dropdown (▾) beside the pills lists the remaining supported languages, alphabetically, to switch to any of them (the chosen one takes the second pill's slot for that session). - Persist the choice in localStorage.ns_locale (shared web ↔ app). Backend values stay canonical English so CAS/analytics are unaffected. - Greek is non-Latin — confirm the chosen fonts include Greek glyph coverage.

Ownership for this feature: - Design — the pill + dropdown UX above, in the compiled bundle; and the professional FR/IT/DE/NL/EL translations for the customer-facing web (app + marketing + prototype), as done for Spanish. - Integration (here) — the same selector logic + OS default in the mobile app; bundling each new language's app strings into the mobile catalogs (a real crossover); extending the founder Translations tool from 2 → 7 languages; integrate.py carries the new locale files and keeps values English-canonical. - Founders — native review/QA of each language before ship. - Decision to lock first — translation source: professional vs machine-translation + native review (recommended: professional / native review for health content).

Skeletons provided: fr.json, it.json, de.json, nl.json, el.json — same key structure as en.json (613 keys incl. mobile), English placeholders. Design/the translator fills the values in place; the mobile side and the founder tool are ready to receive them.

Note on workflow: the marketing site and web app are the compiled bundle Design exports. Each of the above should be fixed in the source design, then re-exported as a normal pack — we re-apply our integration layer (integrate.py) on top. See Change Log → Who fixes what (Design vs Integration) for the ownership split.
