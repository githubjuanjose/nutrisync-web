NutriSync — Platform & Infrastructure Backlog

Scope: production-grade delivery pipeline, scalable containerized backend, secure-zone storage for biometric/personal data, consent everywhere, health & wearable connectors, a unified web entry point, and an internal admin/MIS + protected business-case site.

Legend — Priority: P0 (MVP/launch-blocking) · P1 (fast-follow) · P2 (scale). Estimate: T-shirt (S ≤3d, M ≤1.5wk, L ≤3wk, XL >3wk).

0. Cloud provider recommendation (your question: Cloudflare vs Amazon vs Google)

Recommendation: AWS as the primary cloud, with Cloudflare at the edge in front of it. Use Google Cloud only if the team's deep expertise is there.

Think of it as a layered building: Cloudflare is the security gatehouse and lobby (CDN, WAF, DDoS, bot management, DNS) that every visitor passes through; AWS is the secured building behind it where the regulated data lives in a locked, windowless basement (private subnets + encrypted databases). They are complementary, not either/or.

Why AWS is the core, specifically for NutriSync:

NeedWhy AWS wins for this product
Swiss/EU data residencyAWS has a Zurich region (eu-central-2) plus Frankfurt/Ireland/Paris/Spain. Switzerland is a launch market and menstrual/health data is special-category — keeping it in-country is a strong trust + compliance story.
Regulated health dataMature GDPR + Swiss FADP posture, HIPAA-eligible services, KMS envelope encryption, Nitro isolation, fine-grained IAM, dedicated VPC private subnets, audit via CloudTrail.
Containers + scaleECS Fargate (serverless containers, simplest) or EKS (Kubernetes when you need it), autoscaling, Aurora Postgres in private subnets.
BreadthEverything in one place: managed Postgres, queues (SQS), streams (Kinesis), secrets, observability — fewer moving vendors for a lean team.

Why not the others as the *primary*:

  • Cloudflare is best-in-class at the edge (CDN, WAF, DDoS, Workers, R2, Zero Trust access) and we absolutely use it — but it is not the right home for a regulated relational health-data store with private-subnet isolation. Use it as the front door and for static hosting, not the secure data zone.
  • Google Cloud is excellent (GKE, BigQuery, healthcare APIs) and a fine alternative — but AWS's Zurich region + the broader managed-service surface make it the lower-risk default here. Pick GCP only if the hired engineers are GCP-native.

Net architecture: User → Cloudflare (DNS, CDN, WAF, DDoS, Zero Trust for admin) → AWS ALB → ECS Fargate (API) → Aurora Postgres (private subnet, KMS-encrypted) + S3 (encrypted) + Secrets Manager. Phase-1 can still start on Supabase (EU) for speed (see Architecture doc) and migrate the business logic into ECS as B2B scale demands — the secure-zone principles below apply either way.

EPIC A — Cloud foundation & secure landing zone *(P0)*

Goal: a hardened, multi-environment AWS foundation with a dedicated secure zone for biometric/personal data.

  • A1 (P0, M) — Stand up AWS Organization with separate accounts for dev, staging, prod, and security/audit. *AC: accounts isolated; SСP guardrails deny public S3 and non-EU/CH regions; billing alerts on.*
  • A2 (P0, L) — VPC with public/private subnet tiers in eu-central-2 (Zurich); NAT egress; no public DB. *AC: data stores only in private subnets; security groups least-privilege; flow logs enabled.*
  • A3 (P0, M) — Secure data zone: dedicated private subnet + security group for Aurora Postgres and S3 buckets holding PII/health data; KMS customer-managed keys with rotation. *AC: all PII/health data encrypted at rest with CMK; access only from API security group; key access audited.*
  • A4 (P0, S) — Cloudflare in front: DNS, CDN, WAF, DDoS, rate limiting, bot management. *AC: origin locked to Cloudflare IPs; OWASP ruleset on; TLS 1.2+ end to end.*
  • A5 (P0, S) — Secrets via AWS Secrets Manager (no secrets in repo/CI logs). *AC: app reads secrets at runtime; rotation enabled; CI uses OIDC, no static keys.*
  • A6 (P1, M) — Infrastructure as Code (Terraform) for everything above; remote state + locking. *AC: terraform plan clean; environments reproducible from code.*
  • A7 (P1, S) — Cost guardrails + tagging policy. *AC: per-environment cost dashboards; anomaly alerts.*

EPIC B — CI/CD pipeline: GitHub → production *(P0)*

Goal: every merge flows safely from code to a live, observable production service.

  • B1 (P0, S) — GitHub org + mono-repo, branch protection, required reviews, CODEOWNERS. *AC: no direct pushes to main; PRs require green CI + 1 review.*
  • B2 (P0, M) — CI on GitHub Actions: lint, typecheck, unit tests, build, SBOM, container image scan (Trivy), secret scan. *AC: pipeline fails on critical vulns or leaked secrets; artifacts versioned.*
  • B3 (P0, M) — Container build & push to ECR with immutable tags + image signing (cosign). *AC: only signed images deploy to prod.*
  • B4 (P0, L) — CD: deploy to dev on merge, staging on tag, prod on manual approval (GitHub Environments). Blue/green or rolling via ECS. *AC: zero-downtime deploys; one-click rollback; deploy events logged.*
  • B5 (P0, S) — DB migrations run as a gated pipeline step (expand/contract pattern). *AC: migrations reversible; prod migration requires approval.*
  • B6 (P1, S) — Preview environments per PR (ephemeral). *AC: each PR gets a URL; torn down on merge/close.*
  • B7 (P1, S) — Supply-chain: Dependabot/Renovate + license checks. *AC: weekly dependency PRs; blocked licenses fail CI.*

EPIC C — Scalable containerized backend *(P0)*

Goal: an API platform that scales from 10K to millions of users (per the GTM roadmap).

  • C1 (P0, L) — Containerized API service (NestJS/FastAPI) on ECS Fargate behind an ALB; health checks; autoscaling on CPU/RPS. *AC: scales horizontally; survives instance loss; p95 latency target met under load test.*
  • C2 (P0, M) — Aurora PostgreSQL (Multi-AZ) in the secure zone; PgBouncer/RDS Proxy pooling. *AC: failover < 60s; automated encrypted backups + PITR.*
  • C3 (P0, M) — Object storage (S3, encrypted) for media/exports; signed URLs only. *AC: no public objects; access logged.*
  • C4 (P1, M) — Async layer: SQS + workers for connector syncs, recommendation precompute, notifications. *AC: retries + dead-letter queues; idempotent consumers.*
  • C5 (P1, M) — Caching (ElastiCache/Redis) for phase content, sessions, rate limits. *AC: cache hit ratio target; safe invalidation.*
  • C6 (P2, L) — Recommendation service behind the stable /recommendations contract (rules → ML/Apertus AI). *AC: engine swappable without client changes; inputs/outputs logged (consented, de-identified).*
  • C7 (P2, M) — Multi-region/read-replica readiness for US expansion (2028 roadmap). *AC: documented failover + residency plan.*

EPIC D — Data protection, consent & compliance *(P0 — legally required)*

Goal: biometric/menstrual/health data handled lawfully under GDPR & Swiss FADP, with consent captured everywhere data is collected.

  • D1 (P0, M) — Consent service: versioned consent scopes (account, cycle/health data, wearable sync, research-anon, marketing) with immutable audit trail (who/when/version). *AC: every scope stored with timestamp + policy version; queryable per user.*
  • D2 (P0, M) — Consent in onboarding: explicit, granular, unbundled opt-ins before any health data is stored; can't proceed past health steps without the required consent; plain-language explanations. *AC: no health row persists without a matching consent record; refusing optional scopes still allows core use.*
  • D3 (P0, M) — Consent at point of capture: when connecting a wearable or entering new sensitive data, show purpose + request scope just-in-time. *AC: connector cannot sync until its consent scope is granted; revocation stops sync immediately.*
  • D4 (P0, M) — DSAR endpoints: self-serve data export (machine-readable) and erasure (right to be forgotten), incl. downstream connectors. *AC: export within policy SLA; erasure purges PII across stores + backups schedule documented.*
  • D5 (P0, S) — Row-Level Security / per-user isolation; org admins see only aggregates, never individual cycle data. *AC: cross-user access impossible; verified by tests.*
  • D6 (P0, S) — Data classification + retention policy (special-category vs ordinary); minimization. *AC: each field tagged; retention enforced by jobs.*
  • D7 (P1, M) — DPIA (Data Protection Impact Assessment), Records of Processing, DPA templates for processors. *AC: DPIA signed off before launch; sub-processor list published.*
  • D8 (P1, S) — Cookie/consent banner (privacy-first defaults) on the web entry point. *AC: non-essential off by default; choices logged.*
  • D9 (P1, S) — "Your data is yours" trust surface in-app (visible controls). *AC: users can view/manage/revoke all consents in Profile.*

EPIC E — Health & wearable connectors *(P1 — roadmap 2028, foundation earlier)*

Goal: collect, sync, and securely store activity/health signals from the most-used platforms. Each connector is consent-gated (D3) and normalized to a common schema.

Tier 1 — native phone health platforms (highest reach):

  • E1 (P1, L) — Apple Health (HealthKit) — steps, workouts, sleep, HR, cycle/menstrual data (iOS). *AC: read selected types with user grant; background delivery; revoke handled.*
  • E2 (P1, L) — Google Health Connect / Android (replaces Google Fit) — steps, workouts, sleep, HR. *AC: Health Connect permissions; graceful when app absent.*
  • E3 (P1, L) — Samsung Health — activity, sleep, HR (Android/Galaxy + Watch). *AC: SDK integration; scoped read.*

Tier 2 — wearables / fitness ecosystems:

  • E4 (P1, L) — Garmin Connect (watches) — activity, HR, sleep, stress. *AC: OAuth + webhook pull; backfill window.*
  • E5 (P1, M) — Fitbit — steps, sleep, HR. *AC: OAuth2; rate-limit aware.*
  • E6 (P1, M) — Strava — runs/rides/activities. *AC: OAuth2 + webhooks; dedup vs phone data.*
  • E7 (P2, M) — Oura (ring) — sleep, readiness, HRV, temperature (useful for cycle phase). *AC: OAuth2; temp trends mapped to cycle model.*
  • E8 (P2, M) — WHOOP — strain, recovery, sleep. *AC: OAuth2 + webhooks.*
  • E9 (P2, M) — Polar / Suunto / Withings (pick by demand) — activity/sleep/weight. *AC: at least one of the three; documented.*
  • E10 (P2, M) — Apple Watch / Wear OS companion surfacing — show phase + daily focus on-wrist. *AC: complication/tile reads today's phase + focus.*

Connector platform (shared):

  • E-P1 (P1, L) — Connector framework: normalized health_signal schema, per-provider adapters, token vault, incremental sync + backfill, dedup across overlapping sources. *AC: adding a provider = new adapter only; conflicts resolved by priority rules.*
  • E-P2 (P1, M) — Sync orchestration via SQS workers + webhooks; retry/backoff; observability per provider. *AC: failed syncs alert; per-user sync status visible.*
  • E-P3 (P1, S) — Consent + revoke wiring (depends D3): disconnect purges provider tokens and optionally data. *AC: revoke stops sync in < 1 min; tokens deleted.*

EPIC F — Unified web entry point (one site, not two) *(P0)*

Goal: the commercial/marketing site and the transactional web app are one application behind a single domain, with one entry point that routes anonymous visitors to marketing and authenticated users to the app.

  • F1 (P0, M) — Single web app (Next.js or Expo-Web) serving nutrisynccollective.com: marketing routes public, /app/* authenticated. *AC: one codebase, one deploy, one domain; SEO-friendly marketing pages.*
  • F2 (P0, S) — Smart entry point: visiting root → marketing; logged-in users → redirected to /app; shared nav with "Log in / Get the app." *AC: session-aware routing; deep links preserved.*
  • F3 (P0, M) — Shared auth + session across marketing and app (SSO into the same identity). *AC: single sign-in; waitlist → account upgrade path.*
  • F4 (P0, S) — Shared design system/tokens between marketing, web app, and React Native (Figma-aligned). *AC: one token source; consistent brand.*
  • F5 (P1, S) — Waitlist → onboarding handoff (email capture flows into account + consent). *AC: waitlisted emails convert without re-entry.*
  • F6 (P1, S) — Analytics + A/B on the funnel (EU analytics). *AC: signup funnel measured end-to-end.*

EPIC G — Web admin application (MIS / KPIs) *(P1)*

Goal: an internal admin app (Cloudflare Zero-Trust protected) to run the business — access, subscriptions, and the business case.

  • G1 (P1, M) — Admin app shell with role-based access (admin/finance/clinical/support); behind Cloudflare Access / SSO + MFA. *AC: no public access; roles enforced server-side; all admin actions audit-logged.*
  • G2 (P1, M) — MIS KPI dashboard 4.1: active users / accesses (DAU·WAU·MAU), signups, retention (D1/D7/D30), subscriptions (free vs €5.99 premium, MRR, churn), conversion %, B2B seats, CAC/LTV/ARPU vs targets. *AC: metrics match source-of-truth analytics; date filters; export.*
  • G3 (P1, M) — Business-case / financial tracker (4.1): revenue vs cost vs profit by year, vs the plan in the deck; user-growth actuals vs targets. *AC: editable assumptions; variance vs plan shown.*
  • G4 (P1, S) — Subscription & user admin: search users, view (consented) status, manage entitlements, handle refunds/escalations (no access to raw cycle data). *AC: PII access role-gated + logged; cycle data never shown individually.*
  • G5 (P1, S) — Connector/ops health panel: per-provider sync status, error rates. *AC: surfaces E-P2 metrics.*
  • G6 (P2, S) — Content/clinical review queue (phase content + recipes evidence-tagging). *AC: publish workflow with clinical sign-off.*

EPIC H — Protected business-case site *(P1)*

Goal: an access-controlled site presenting the full business case, pitch, and documentation (from the deck/Canva), for investors/advisors/team only.

  • H1 (P1, M) — Business-case microsite built from the pitch deck: problem, solution, market (FemTech boom), differentiation, traction, business model, GTM, financials, team, advisory. *AC: all deck sections represented; responsive; brand-consistent.*
  • H2 (P1, S) — Access protection via Cloudflare Access (email-allowlist / SSO) — real auth, not client-side. *AC: only invited emails can view; access logged; revocable.*
  • H3 (P1, S) — Document vault: downloadable deck (PDF), financial model, one-pager, data-room links. *AC: gated downloads; per-file access tracked.*
  • H4 (P2, S) — Embed the live Canva presentation + downloadable export. *AC: deck viewable inline once Canva export/link provided.*
  • H5 (P2, S) — Per-visitor view analytics (which investor viewed what). *AC: per-email engagement visible in admin.*

A static preview of H1 (built from the deck) ships with this backlog as web/business-case/index.html with a placeholder gate — replace the gate with H2 (Cloudflare Access) before sharing externally.

  3. Wave 3 (scale, P1→P2): E4–E10, C6–C7, G5–G6, H4–H5, I5.

Dependencies & risks

  • D (consent) blocks E (connectors) — never sync a wearable without its consent scope.
  • A3 (secure zone) blocks C2/C3 (data stores) — provision the secure zone before any PII/health data lands.
  • F3 (shared auth) blocks G/H protection — one identity system feeds admin + business-case access.
  • Apertus AI integration model (hosted vs self-run; residency/latency/cost) — decision needed before C6.
  • Connector ToS/rate limits (Garmin, WHOOP, Strava) and Apple/Google review policies — legal + review lead time.
