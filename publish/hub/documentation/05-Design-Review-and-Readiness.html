NutriSync — Design Review & Operational-Readiness Checklist

Based on a review of the Figma file (Dev Mode MCP) covering ~10 screen sections and component/variant boards. This documents what the designs contain, observations and gaps, and a concrete list of everything required to make the app fully operational.

1.2 Design system observations

  • Per-phase color system (important): menstrual = red/coral, follicular = green, ovulatory = gold/yellow, luteal = pink/magenta. This is core brand IP and must be a first-class token set across web + RN.
  • Dual scoring concept: "Cycle Sync Score" (home) and "Cycle Alignment Score" (progress) — clarify whether these are the same metric named differently. Naming consistency needed.
  • Secondary green = "good/healthy score" state (rings, deltas), paired with the coral brand.
  • Rich data-viz: rings, line charts, area charts, horizontal bar charts, contributor tags. Needs a charting approach (e.g. Victory Native / React Native SVG; Recharts on web).
  • Bottom navigation: Calendar · Cycle · Movement · Progress (4 tabs). NutriLog/MovementLog accessed from the cycle/home context. (Current prototypes use Home/Meals/Progress/Community/Profile — IA needs realigning to the Figma.)
  • Component/variant boards exist (avatar colors, buttons, consent cards, phase-timeline variants, NutriLog atoms) but no formal token frames (colors/type) — tokens are implicit on nodes.

1.3 Strengths

  • Cohesive, warm, and distinctive brand; the per-phase color language is a strong differentiator.
  • Genuinely useful, science-led IA (cycle-synced nutrition + movement + scoring).
  • Onboarding already includes a consent step — good privacy instinct.

1.4 Gaps & risks to resolve in design

  • No explicit token library (colors/typography/spacing) → create Figma Variables/Styles so design and code share one source of truth.
  • Score definitions undocumented — how are Cycle Alignment/Sync, Mood, Energy scores calculated? Needs a spec (inputs, formula, ranges, what "good" means). This is essential IP and blocks engineering.
  • Empty / error / loading states not visible for most screens (e.g. no data yet, sync failed, offline, no connector).
  • Accessibility: verify text contrast on coral/gradient backgrounds (WCAG AA), min 44px touch targets, dynamic type, and that phase meaning isn't conveyed by color alone (add labels/icons).
  • Localization: Switzerland + Spain launch → needs EN/ES (and likely DE/FR for CH) — design for longer strings.
  • Content placeholders: "Maria/Sarah/Olivia", lorem tips, and sample nutrients need finalized, clinically-reviewed content.
  • Edge data: very long cycles, irregular cycles, pregnancy/menopause, contraceptive users (already a flow) — define how phase logic and scores behave.

Part 2 — What's needed to make the app FULLY operational

Grouped by area. Items marked [BLOCKER] gate a usable launch.

2.1 Design assets & specs (from design)

  • ☐ [BLOCKER] Figma token library: colors (incl. 4 phase colors + green score states), typography scale, spacing, radii, elevation — published as Variables/Styles.
  • ☐ [BLOCKER] Exported icon set and the NutriSync logo/wordmark + app icon + splash (all densities).
  • ☐ Component specs with states (default/pressed/disabled/loading/error/empty) for buttons, cards, list rows, chips, score rings, charts.
  • ☐ Empty/error/loading/offline designs for each screen.
  • ☐ Onboarding flow final copy + the consent screen wording (legally reviewed).
  • ☐ Notification designs + copy (mood/energy prompts, daily focus, sync results).
  • ☐ Accessibility pass (contrast, touch targets, color-independent phase cues, dynamic type).
  • ☐ Responsive/web adaptations of the mobile screens (for the unified web app).

2.2 Product/algorithm specs (the IP)

  • ☐ [BLOCKER] Score definitions: Cycle Alignment Score, Cycle Sync Score, Mood Score, Energy Score — inputs, formula, range, update cadence, "contributors/areas to improve" logic. (Clarify if Sync == Alignment.)
  • ☐ [BLOCKER] Phase model spec: how phases are derived/personalized (incl. irregular cycles, contraceptive users, data from wearables like Oura temperature).
  • ☐ [BLOCKER] Cycle-synced content matrix per phase: nutrition tips, Nutri Basics nutrients, movement tips, recipes — clinically reviewed + evidence-tagged.
  • ☐ Recommendation logic spec (rules for MVP → Apertus AI later) behind the /recommendations contract.
  • ☐ Recipe/nutrient content library (licensed, original, or partner) with macros + sources.

2.3 Backend & data (see Architecture + Platform Backlog)

  • ☐ [BLOCKER] Auth & accounts (email/social), session, password/security (Settings → sign-in & security exists in design).
  • ☐ [BLOCKER] Database schema for: profile, cycle profile, cycle/mood/energy logs, nutri/movement logs, scores, recipes, nutrients, consents, connectors, notifications. (Initial schema already drafted in supabase/migrations/0001_init.sql — extend for scores/logs/connectors.)
  • ☐ [BLOCKER] Secure data zone (encrypted, private subnet) for health/biometric data — Epic A.
  • ☐ APIs for each screen's reads/writes (home, nutrilog, movementlog, progress/scores, calendar, settings, notifications).
  • ☐ Score computation jobs (async) + history storage for the charts.
  • ☐ Notifications service (push) for mood/energy/daily focus.
  • ☐ [BLOCKER] Consent service + consent capture in onboarding and at point of data capture — Epic D.
  • ☐ DSAR: data export + erasure — Epic D.

2.8 Operations

Part 3 — Top blockers to "fully operational" (do these first)

  1. Score & phase algorithm specs — the product can't function without defined Cycle Alignment/Sync, Mood, Energy logic.
  2. Clinically-reviewed cycle-content matrix (nutrition, nutrients, movement, recipes).
  3. Design token library + states/empty-error designs.
  4. Auth + secure data zone + consent service (legal prerequisite for storing health data).
  5. IA realignment + signature-screen rebuild to the real Figma (web + RN).
  6. GDPR/FADP legal pack (DPIA, policies, consent wording).

Everything else (connectors, admin app, business-case site, full CI/CD) layers on top of these and is captured in 03-Platform-Backlog.md / backlog.csv.
